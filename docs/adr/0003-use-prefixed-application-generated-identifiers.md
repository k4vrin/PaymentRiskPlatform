# ADR 0003: Use Prefixed Application-Generated Identifiers

## Status

Accepted

## Date

2026-05-26

## Context

The payment platform creates identifiers for several kinds of records:

- payment aggregates;
- payment authorization rows;
- payment risk decision rows;
- idempotency records;
- outbox events;
- correlation IDs.

Before this ADR, identifier generation was split across domain objects, mappers, and filters. That made it harder to
answer basic questions such as which prefix belongs to which record, whether IDs are generated by the application or the
database, and how new record types should choose an identifier format.

The project needs IDs that are:

- unique enough for local and distributed service execution;
- readable in logs, APIs, database rows, and events;
- portable across PostgreSQL and Oracle-compatible schemas;
- generated before persistence when IDs must be referenced by idempotency records, outbox events, or API responses;
- easy to validate in tests.

## Decision

Use application-generated, prefixed identifiers for platform records.

Introduce a central `PlatformIdGeneratorFactory` in the Java service shared package. The generator owns the current
prefixes
and generation format:

| Entity or concept            | Prefix/format              |
|------------------------------|----------------------------|
| Payment aggregate ID         | `pay_` + UUID string       |
| Payment authorization row ID | `pauth_` + compact UUID    |
| Payment risk decision row ID | `prd_` + compact UUID      |
| Idempotency record ID        | `idem_rec_` + compact UUID |
| Outbox event ID              | `evt_` + compact UUID      |
| Correlation ID               | UUID string                |

Use canonical UUID strings for externally visible aggregate-style IDs where readability is useful. Use compact UUIDs
without hyphens for internal row/event IDs where the prefix already explains the type and shorter values are easier to
scan in tables.

Authorization codes are not platform record identifiers. They remain a separate business code generated with
`SecureRandom` and the `AUTH-` prefix.

## Consequences

Application code can create IDs before database insert. This is useful when a payment ID must be returned to the API
client, written into an idempotency record, included in outbox events, or passed to the risk service.

Prefixing makes logs and database rows easier to inspect. A developer can usually tell whether `pay_...`, `pauth_...`,
`prd_...`, `idem_rec_...`, or `evt_...` refers to a payment, authorization row, risk decision row, idempotency record,
or event.

The tradeoff is that the application owns uniqueness behavior instead of delegating it to database sequences. The
current UUID-based strategy is acceptable for this project because it avoids database-specific sequence behavior and
keeps the schema portable across PostgreSQL and Oracle-compatible deployments.

If the project later needs time-sortable IDs, the generator can be changed behind the central factory to use ULID,
UUIDv7, or another monotonic format. That change should be covered by a follow-up ADR because it affects ordering,
index locality, and external API expectations.

## Rejected Alternatives

- Database sequences for all IDs: rejected because they are less portable across local PostgreSQL and
  Oracle-compatible targets, and they make it harder to create IDs before persistence.
- Plain UUIDs without prefixes: rejected because they are harder to inspect in logs, events, and operational tooling.
- One global `id_` prefix for every record: rejected because it loses useful type information.
- Reusing authorization codes as identifiers: rejected because authorization codes are business-facing approval codes,
  not primary keys or platform record identifiers.
- Random numeric IDs: rejected because they add collision and formatting concerns without improving readability or
  portability.
